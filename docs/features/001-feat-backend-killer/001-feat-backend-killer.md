# User Spec: backend-killer

## Что делаем

Новая harmful-нагрузка noisia — `backend-killer`. Одна-единственная PostgreSQL-сессия
постепенно, под управляемым клиентом темпом, раздувает память своего backend-процесса за
счёт утечки prepared statements (рост plan cache): сессия непрерывно создаёт уникальные
`PREPARE`, план-кэш backend'а растёт, RSS процесса лезет вверх — пока OOM killer не убьёт
backend, что postmaster трактует как сбой и перезапускает весь инстанс. Классический эффект
«одна сессия положила всю базу».

Нагрузка однопоточная, без предзасева данных (нагрузка самогенерируется), с «панелью
эскалации» в логах (счётчик prepared statements + темп + время), которая по принципу
самоотчёта переживает крах БД.

## Зачем

Учебный материал для ведущих DBA, обучающих младших. На стенде одной командой
воспроизводится сценарий «рост RSS backend'а → OOM → перезапуск инстанса», который иначе
ловится только в проде или хрупкими ad-hoc скриптами. Ведущий показывает джунам поведение
системы под экстремальным давлением на память, разбирает симптомы (OOM в логах ядра,
перезапуск PostgreSQL, рост plan cache) и варианты решения. Вторичная ценность — проверка
срабатывания мониторинга и алертов (память, рестарт инстанса).

`backend-killer` закрывает уникальный для noisia слот: ни одна из 8 существующих нагрузок не
доводит инстанс до OOM и перезапуска от действий одной сессии (`idlexacts` даёт bloat,
`tempfiles` — disk spill/IO, но не OOM).

## Пользовательские истории

- Как ведущий DBA, я хочу одной командой запустить `backend-killer` на учебном стенде, чтобы
  показать джунам, как одна сессия с утечкой prepared statements доводит инстанс до OOM и
  перезапуска, и разобрать симптомы и решения.
- Как SRE / инженер мониторинга, я хочу по запросу воспроизвести рост памяти backend'а с
  последующим OOM-перезапуском, чтобы убедиться, что алерты срабатывают (память, рестарт
  инстанса, рост plan cache).

### Пользовательские сценарии

**Сценарий 1: Демонстрация OOM на учебном стенде (ведущий DBA)**
1. Ведущий запускает `noisia --backend-killer --conninfo="..."` против стенда с
   ограниченной памятью (контейнер/VM, обычно co-located с noisia).
2. noisia открывает одно выделенное соединение и начинает непрерывно слать уникальные
   `PREPARE` (с автоподбором «тяжести» плана под объём памяти стенда).
3. Каждые `report-interval` (по умолчанию 1s) в лог пишется панель эскалации:
   `backend-killer: prepared stmts=N rate=R/s elapsed=T` (опционально с оценкой памяти
   backend'а при `--show-memory`).
4. RSS backend-процесса растёт → OOM killer убивает backend → postmaster перезапускает весь
   инстанс → соединение noisia рвётся.
5. noisia ловит обрыв своего соединения и пишет финальную строку:
   `connection lost after T, N statements issued — target likely OOM-restarted`.
6. Результат: ведущий на живом примере показывает симптомы (OOM в dmesg/логе ядра, рестарт
   PostgreSQL в его логе, рост plan cache) и обсуждает решения (контроль prepared statements,
   `DEALLOCATE`, лимиты памяти, поведение пулеров).

**Сценарий 2: Проверка мониторинга (SRE)**
1. SRE запускает `backend-killer` против тестового инстанса, подключённого к мониторингу.
2. По мере роста памяти и последующего перезапуска инстанса наблюдает срабатывание алертов.
3. Результат: подтверждено, что алерты на давление памяти и рестарт инстанса работают.

## Дизайн и интерфейс

CLI-нагрузка, без UI. Интерфейс — флаги командной строки и периодический вывод в лог.

### CLI-флаги

- **`--backend-killer`** — включить нагрузку (bool).
- **`--backend-killer.rate`** — троттлинг создания prepared statements, шт/сек (float).
  По умолчанию `0` = без лимита.
- **`--backend-killer.plan-size`** — «тяжесть» одного плана (int), то есть сколько памяти
  тратит каждый `PREPARE` (управляется числом выражений в target-list плана).
  По умолчанию `0` = автоподбор: noisia определяет доступную память (сначала лимит cgroup,
  затем `/proc/meminfo`) и подбирает значение так, чтобы OOM наступил за разумное время демо.
  Любое значение `>0` — явный override.
- **`--backend-killer.size-multiplier`** — множитель к автоподбору (float), по умолчанию `1.0`.
  Позволяет ускорять/замедлять демо, не оперируя абсолютными числами.
- **`--backend-killer.show-memory`** — добавить в панель оценку памяти своего backend'а (bool),
  через `SELECT sum(used_bytes) FROM pg_backend_memory_contexts` (PG 14+). По умолчанию выкл.
- **`--backend-killer.report-interval`** — период вывода панели (duration), по умолчанию `1s`.

Каждый флаг имеет `.Default()` и `.Envar("NOISIA_...")`, как у остальных нагрузок.
Флаг `--jobs` не применяется — нагрузка принципиально однопоточная.

### Формат вывода (панель эскалации)

Строка раз в `report-interval` через `logger.Infof`:
```
backend-killer: prepared stmts=124500 rate=8200/s elapsed=15s
```
С `--show-memory`:
```
backend-killer: prepared stmts=124500 rate=8200/s elapsed=15s backend mem≈3.1GB
```
Финальная строка при обрыве собственного соединения (кульминация):
```
connection lost after 4m20s, 2128000 statements issued — target likely OOM-restarted
```
Прогресс-бара в процентах нет: предел задаётся железом и в модели самоотчёта неизвестен.

### UX-поведение

- Самоотчёт — основной канал: счётчик prepared statements ведётся в процессе noisia и точно
  известен без опроса сервера, поэтому панель информативна даже когда БД уже недоступна.
- Единственный запрос к серверу за рамками `PREPARE` — опциональное чтение памяти **своего**
  backend'а при `--show-memory` (та же сессия, не внешний мониторинг).
- Остановка — только по отмене context (`--duration` или Ctrl+C): нагрузка завершается чисто.

## Как должно работать

### Основной сценарий
1. noisia открывает **одно выделенное соединение** (`db.Connect`, не пул — пул размазал бы
   `PREPARE` по разным backend'ам, и ни один не дошёл бы до OOM).
2. Если `plan-size=0` — определяет доступную память (cgroup-лимит → `/proc/meminfo`) и
   вычисляет «тяжесть» плана, масштабируя на `size-multiplier`.
3. В цикле непрерывно отправляет уникальные `PREPARE` (имя — из монотонного счётчика),
   соблюдая `rate` (если задан); счётчик созданных statements растёт монотонно.
4. Раз в `report-interval` печатает панель эскалации (+ память backend'а при `--show-memory`).
5. При OOM соединение рвётся → noisia печатает кульминационную строку и завершается.

### Граничные случаи
- **Стенд с большим RAM:** OOM может наступать долго → автоподбор/`plan-size`/`size-multiplier`
  повышают расход памяти на каждый `PREPARE`.
- **Уникальность имён `PREPARE`** на миллионах statements — генерируются из монотонного счётчика.
- **context отменён до OOM** (типичный случай в CI/тестах) — нагрузка просто чисто
  останавливается (возврат `nil`).
- **Инстанс перезапустился и вернулся** — noisia уже напечатала кульминацию и завершилась;
  цикла переподключения нет.
- **`rate=0` (без лимита)** не должен патологически крутить busy-loop — используется та же
  идиома цикла, что в `rollbacks`/`tempfiles`.
- **Автоподбор и co-location:** procfs/cgroup измеряют машину, где запущена **noisia**, а не
  сервер PostgreSQL. Автоподбор корректен только когда noisia co-located с PG (типичный стенд).
  Для удалённого PG пользователь должен задать `--backend-killer.plan-size` явно.
- **Память в контейнере:** внутри контейнера `/proc/meminfo` показывает RAM хоста; реальный
  потолок — в cgroup. Автоподбор сначала читает cgroup-лимит (v2 `/sys/fs/cgroup/memory.max`,
  fallback v1), затем `/proc/meminfo`.
- **`--show-memory` на PG < 14:** `pg_backend_memory_contexts` отсутствует → деградировать
  мягко (пропустить строку памяти / предупредить один раз), счётчик оставить.

## Критерии приёмки
- [ ] Флаг `--backend-killer` включает нагрузку; присутствуют флаги `--backend-killer.rate`
      (0=без лимита), `--backend-killer.plan-size` (0=auto), `--backend-killer.size-multiplier`
      (1.0), `--backend-killer.show-memory`, `--backend-killer.report-interval` (1s) — с
      дефолтами и envars.
- [ ] Нагрузка использует **одно выделенное соединение** (`db.Connect`), не пул.
- [ ] Непрерывно создаются уникальные `PREPARE`; внутренний счётчик растёт монотонно.
- [ ] «Тяжесть» плана зависит от `plan-size`; при `plan-size=0` происходит автоподбор по
      доступной памяти (cgroup → `/proc/meminfo`), масштабируемый `size-multiplier`;
      `plan-size>0` — явный override.
- [ ] `rate` троттлит шт/сек при значении `>0`; `0` = без лимита.
- [ ] Панель эскалации пишется раз в `report-interval` через `logger.Infof`; опроса состояния
      **сервера** нет. При `--show-memory` добавляется оценка памяти своего backend'а
      (`pg_backend_memory_contexts`, PG 14+; на PG < 14 — мягкая деградация).
- [ ] При обрыве собственного соединения (OOM) и активном context печатается кульминационная
      строка; нагрузка завершается с `nil` (это успех реального прогона, не ошибка).
- [ ] Нагрузка чисто останавливается по отмене/таймауту context (возврат `nil`).
- [ ] `Conninfo` нигде не логируется.
- [ ] Интеграционный тест проверяет механику (монотонный рост счётчика за N итераций +
      чистая остановка по context), **без** доведения до реального OOM.
- [ ] `Config.validate()` покрыт table-тестом.
- [ ] README обновлён (bullet-список нагрузок + таблица impact, impact = **Yes**).

## Ограничения

- Следовать конвенциям пакета-нагрузки noisia: отдельный Go-пакет, `Config` + `validate()`,
  `NewWorkload`, `Run(ctx)`; доступ к драйверу — через интерфейсы пакета `db` (pgx v5);

- Самоотчёт — основной канал (счётчик переживает крах). Единственный допустимый запрос к
  серверу сверх `PREPARE` — опциональное чтение памяти **своего** backend'а при `--show-memory`.
- Автоподбор `plan-size` — эвристика для разумного времени демо, **не** гарантия точного
  момента OOM (мешают overcommit, `shared_buffers`, сторонние процессы); для контроля есть
  `size-multiplier` и явный override.
- Разрушительная нагрузка: может уронить реальный инстанс. Сохраняется позиция проекта —
  без защитных «предохранителей», ответственность на пользователе (дисклеймер в README).
- Только PostgreSQL.

## Риски
- **Разрушительность:** может обрушить реальный инстанс. **Митигация:** дисклеймер в README +
  запись в таблице impact (impact = Yes); позиция проекта «без guardrails» сохраняется.
- **Небезопасность для CI:** реальный OOM убил бы контейнер и сделал тесты флакающими.
  **Митигация:** тесты проверяют только механику, реальный OOM в CI не запускается.
- **Ошибочный автоподбор `plan-size`** (удалённый PG; cgroup нечитаем): демо может стать
  слишком быстрым/медленным. **Митигация:** override `--plan-size` + `--size-multiplier`;
  документировать требование co-location; предупреждать при fallback на `/proc/meminfo`.
- **Утечка секрета:** **Митигация:** `Conninfo` никогда не логируется.
- **`--show-memory` на старом PG:** **Митигация:** на PG < 14 мягко пропустить строку памяти.

## Технические решения
- Мы решили раздувать память **инкрементально в одной сессии** (а не одним блокирующим
  «тяжёлым» запросом и не множеством сессий), потому что только так «одна сессия → OOM»
  совмещается с живой панелью эскалации.
- Мы решили использовать конкретно **утечку prepared statements (рост plan cache)** как
  триггер: это узнаваемый реалистичный путь к росту RSS → OOM (ORM/драйвер/пулер без
  `DEALLOCATE`). Сценарий теоретический (не из личного инцидента автора), но сам класс
  «рост RSS → OOM» — хорошо известный из практики.
- Мы решили **не** воспроизводить OOM через `work_mem`-сортировки/hash join, потому что они
  обычно дают disk spill + IO (а не OOM), и эта область уже покрыта `tempfiles`.
- Мы решили вести **самоотчёт** (счётчик prepared statements + темп + время) как основной
  канал, потому что он точно известен noisia и переживает крах БД, когда мониторинг состояния
  слепнет.
- Мы решили добавить **опциональную** строку памяти своего backend'а (`--show-memory`), чтобы
  в учебном демо был виден не только «причинный» счётчик, но и «следствие» — рост памяти;
  это чтение своего backend'а, а не внешний мониторинг.
- Мы решили использовать **одно выделенное соединение** (`db.Connect`), не пул, потому что пул
  размазал бы `PREPARE` по разным backend'ам и ни один не дошёл бы до OOM.
- Мы решили сделать нагрузку **однопоточной** (без `--jobs`) и **без предзасева** данных.
- Мы решили определять кульминацию через **обрыв собственного соединения** noisia.

## Тестирование

**Unit-тесты:** делаются всегда. Table-тест `Config.validate()` (валидные/невалидные комбинации).

**Интеграционные тесты:** делаем — проект integration-heavy. Под testcontainers
(`postgres:15-alpine` через `dbtest.RunMain`) проверяем механику: соединение открывается,
счётчик растёт монотонно за N итераций, нагрузка чисто останавливается по отмене context.
Реальный OOM в тестах **не** запускается. Тесты сериализуются (`-p 1`).

**E2E тесты:** не делаем — для CLI-утилиты отдельный E2E-слой избыточен; интеграционных
тестов + ручной проверки на стенде достаточно.
